# 하네스 작업 가이드

## 전체 흐름

```
request 작성  →  "계획생성"  →  work 검토  →  "시작"  →  구현 완료
   (사람)            (Claude)       (사람)       (사람)     (Claude)
```

사람이 직접 해야 할 단계는 **request 작성**과 **work 검토** 2개.

---

## 1. Request 작성 방식

### 파일 위치 및 명명

```
plans/<domain>/request/plan-<name>-request.md
```

---

### 필수 항목

#### 공통
```
<기능 설명>
- 인증x / 인증o
- <HTTP METHOD> <endpoint>
```

#### 파라미터
```
- body / params / query
    - <필드명>: <타입> => <이 도메인에서의 의미와 예시>
```

#### 처리 흐름
```
- 처리 흐름
    - 각 단계를 순서대로 기술
    - 분기 조건 명시 (실패해도 진행 / 실패시 중단 등)
    - 저장 여부 명시 (DB insert 없이 발송만 등)
```

---

### 작성 원칙

**명시하지 않으면 Claude가 임의로 채운다.**

빠뜨리면 안 되는 것들:
- 각 필드의 **이 도메인에서의 의미** — 필드명만 쓰면 다른 도메인 설명을 재사용할 수 있음
- **분기 조건** — 조회 실패 시 계속 진행인지, 중단인지
- **저장 여부** — 발송만 할지, DB에도 남길지
- **고정값** — 코드나 타입 등 변수로 받지 않고 고정해야 하는 값

---

### 하네스 위배 주의사항

아래 방식은 Claude의 판단에 의존하게 되어 하네스 원칙에 위배된다.

| 위배 패턴 | 이유 | 올바른 방식 |
|-----------|------|-----------|
| "기존 기능과 같은 로직으로 구현" | 기존 코드 스캔 후 Claude가 패턴을 임의 해석 | 처리 흐름 전체를 request에 직접 명시 |
| 다른 도메인/액션 파일 참조 | request 외부 정보에 의존 | request 안에 모든 값을 완결되게 작성 |
| 필드 설명 생략 | Claude가 유사 도메인 설명을 재사용 | 필드마다 이 도메인의 맥락에 맞는 설명 작성 |


---

## 2. Work 파일 검토

### 검토 명령

```
"<파일명> 계획생성" → work.md 생성됨 → 검토 후 "시작"
```

---

### 검토 체크리스트

#### 엔드포인트
- [ ] HTTP 메서드가 request와 일치하는지
- [ ] URL 경로가 request와 일치하는지
- [ ] path param / query / body 위치가 맞는지

#### DTO 필드 설명
- [ ] 각 필드의 description이 이 도메인 맥락에 맞는지
  - 다른 도메인 작업 내용이 그대로 복사된 건 아닌지

#### 처리 흐름
- [ ] 분기 조건이 request와 일치하는지
- [ ] 저장/발송 여부가 request와 일치하는지
- [ ] 고정값이 올바르게 반영됐는지

---

### work 검토의 핵심

request에 적은 것과 work에 나온 것이 **1:1로 대응**되는지 확인.
대응되지 않는 항목이 있으면 "시작" 전에 수정 요청.
work가 틀렸다면 request를 먼저 수정하고 재생성.
